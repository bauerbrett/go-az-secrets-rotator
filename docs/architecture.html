Core Rotation Flows
1 Scheduled Rotation (Happy Path)
Scheduler polls DB — finds secret with next_rotation_at <= NOW() and active policy
Scheduler creates rotation_jobs row (PENDING) — snapshots max_attempts from policy
Scheduler checks worker is APPROVED + last heartbeat < 5 min ago
Scheduler publishes message to tenant's Service Bus topic → job PUBLISHED
Worker receives message from subscription, reports RUNNING via API callback
Worker executes rotation plugin (create new credential on target resource)
Worker stores new secret version in tenant's Key Vault
Worker validates credential if validation_required = true (policy)
Worker reports COMPLETED → API resets next_rotation_at = NOW() + rotation_interval
2 Manual Rotation (On-Demand)
Tenant Admin calls POST /api/tenants/{tid}/secrets/{sid}/rotate
API creates rotation_job (PENDING) with trigger = 'manual'
Scheduler picks up PENDING job on next publish loop (≤60s)
Same flow as scheduled from step 4 onward
3 Worker Onboarding
Worker starts → acquires MI token from Azure AD (client_credentials)
Worker calls POST /api/workers/register through APIM with JWT
APIM validates JWT (issuer, audience, MI object ID claim)
API creates worker record with status = PENDING_APPROVAL
Tenant Admin reviews and approves → status = APPROVED
API grants Service Bus Data Receiver on tenant subscription
Worker subscribes to rotation messages + starts heartbeats (60s interval)
Policy Flows
4 Policy Creation & Assignment
Tenant Admin creates policy: POST /api/tenants/{tid}/policies — rotation_interval, max_retry, backoff_base, validation_required
API validates secret_lifetime > rotation_interval (enforces safety buffer)
Tenant Admin assigns policy to secret: PATCH /api/tenants/{tid}/secrets/{sid}
API computes next_rotation_at = NOW() + rotation_interval
Scheduler now sees this secret in its polling query when rotation is due
5 Policy Evaluation by Scheduler
Scheduler JOINs secret_registrations + rotation_policies on policy_id
Filters: status=ACTIVE, policy assigned, next_rotation_at <= NOW(), no active job
Snapshots max_retry_attempts + validation_required into job (immutable for job lifetime)
retry_backoff_base NOT copied — read live so policy updates take immediate effect
After COMPLETED, API resets next_rotation_at using current policy interval
6 Policy Update Behavior
Admin updates policy (e.g., rotation_interval 30d → 7d)
Existing secrets: no recomputation of next_rotation_at (avoids thundering herd)
In-flight jobs: unaffected — max_attempts already snapshotted
retry_backoff_base: immediate effect (read live by Retry Promoter)
Next natural rotation cycle uses new interval going forward
System Task Flows
7 Job Reconciliation (every 5 min)
Job Reconciler finds PUBLISHED jobs > 1 hour old → resets to PENDING (publish was lost/expired)
Finds RUNNING jobs > 30 min old → marks FAILED ("worker unresponsive")
Finds VALIDATING jobs > 15 min old → marks FAILED (validation hung)
Each state change writes audit_logs entry with reason code
8 Retry Promotion (every 5 min)
Retry Promoter finds FAILED jobs where attempt < max_attempts
Calculates backoff: base × 2^(attempt-1) → e.g. 5m, 10m, 20m
If past backoff window: resets to PENDING, increments attempt counter
Scheduler picks up re-PENDING'd job in next publish loop → retry begins
9 Dead-Letter Promotion (every 5 min)
Dead-Letter Promoter finds FAILED jobs where attempt >= max_attempts
Sets status DEAD_LETTERED — job is now terminal
Audit log: job.dead_lettered (future: tenant notification)
Requires manual intervention — admin must investigate and re-trigger
10 DLQ Drainer (every 5 min)
DLQ Drainer peeks Service Bus dead-letter queues per tenant (up to 50 msgs)
Extracts job_id from message properties → looks up job in DB
If not already terminal → marks FAILED with DLQ reason
Completes (removes) DLQ message — Retry Promoter handles rest
Alerts if DLQ depth > 20 (worker may be broken)
11 Worker Health Monitor (every 2 min)
Health Monitor finds APPROVED workers with last_heartbeat > 5 min ago
Sets worker SUSPENDED → no new jobs published to it
Removes Service Bus Data Receiver RBAC from worker's MI
In-flight RUNNING job caught by Job Reconciler (30 min threshold)
Self-heal path: worker resumes heartbeats → admin re-approves → RBAC restored
12 Cleanup Tasks (daily)
Heartbeat Cleanup (02:00 UTC): purges worker_heartbeats > 30 days
Audit Log Cleanup (03:00 UTC): purges audit_logs > 90 days
If partitioned → DROP TABLE (instant). Otherwise batched DELETE (LIMIT 10000)
Security Architecture & Controls

Identity & Authentication

  • Workers: Azure Managed Identity → client_credentials JWT
  • Users: Azure AD interactive login (PKCE authorization code)
  • APIM validates JWT at the edge (issuer, audience, expiry, claims)
  • APIM injects its own MI token as X-Gateway-Auth (defense-in-depth)
  • Backend verifies BOTH caller JWT + gateway MI token
  • No shared secrets — all identity is MI/certificate-based

Network Security

  • Dedicated VNET (10.0.0.0/16) — AKS subnet + APIM subnet
  • Cilium CNI with default-deny across all namespaces
  • Explicit allow rules only for documented flows
  • L7 HTTP policies for path/method control
  • PostgreSQL via private endpoint only (no public access)
  • Service Bus accessed via MI RBAC (no connection strings)
  • Workers in separate customer VNETs (network isolation)

Authorization & Tenant Isolation

  • PostgreSQL Row-Level Security (RLS) — all queries scoped to tenant_id
  • System Tasks bypass RLS via az_rotator_system role
  • Service Bus topic-scoped RBAC — workers access own tenant topic only
  • Platform RBAC via Azure AD App Roles (TenantAdmin, Operator)
  • Worker approval gate — admin must approve before access
  • Policy/secret CRUD scoped to tenant membership

Pod Security (All Services)

  • runAsNonRoot: true (UID 65534 in distroless)
  • allowPrivilegeEscalation: false
  • capabilities.drop: ["ALL"]
  • readOnlyRootFilesystem: true
  • Resource requests/limits enforced
  • No service account token auto-mount
  • distroless/static — no shell, no package manager

Secrets Management

  • External Secrets Operator (ESO) syncs from Key Vault
  • K8s Secrets as runtime delivery (never env vars or configmaps)
  • Key Vault access via Workload Identity (no static creds)
  • Auto-refresh on configurable interval
  • Worker credentials never transit the control plane
  • Tenant Key Vaults are customer-owned (MI RBAC)

Queue Poisoning Prevention

  • Only Scheduler MI has Service Bus Data Sender role
  • Workers: Data Receiver on their subscription only
  • Topic-scoped RBAC prevents cross-tenant injection
  • Cilium blocks pod→SB egress except Scheduler namespace
  • No shared keys or connection strings — all MI-based
  • DLQ Drainer reconciles poisoned messages to DB state

Audit & Observability

  • Append-only audit_logs table (partitioned, 90-day retention)
  • Structured JSON logging with correlation IDs
  • OpenTelemetry → App Insights (traces + dependencies)
  • Log Analytics for centralized KQL queries
  • All system task actions logged with reason codes
  • Grafana dashboards for rotation SLAs and health

Defense in Depth Layers

  • Edge: APIM JWT validation + rate limiting
  • Network: VNET isolation + Cilium default-deny + private endpoints
  • Transport: TLS everywhere (HTTPS, encrypted Service Bus)
  • Application: Dual-token verification (caller + gateway)
  • Data: RLS, encryption at rest, private endpoint access
  • Operational: Worker approval gate, RBAC separation, CronJob hygiene
